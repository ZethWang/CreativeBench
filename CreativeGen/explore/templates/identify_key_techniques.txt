### Example Analysis

For a sorting problem using **quicksort**:

**Core Technique: Recursive Divide-and-Conquer**
  Quicksort relies on recursion to divide arrays into subproblems around a pivot and sort them recursively.

**Level 1 Constraint: No Built-in Sort Functions**
  Prevents easy shortcuts (`sorted`, `.sort`), requiring a custom approach but still allowing familiar patterns.

**Level 2 Constraint: No Recursion**
  Forces the use of an iterative method (e.g., explicit stack), pushing beyond the default recursive pattern.

**Level 3 Constraint: No Explicit Loops**
  Without `for` or `while`, the model must adopt functional tools (e.g., `map`, `reduce`), exploring less conventional solutions.

## Constraint Design Principles
1. **Individually Reasonable**: Each constraint blocks one key technique but leaves viable alternatives.
2. **Cumulative & Solvable**: Level k must include all constraints from levels < k. Ensure the combined set at levels 6–7 is still solvable for the given problem.
3. **No Trivial Workarounds**: Avoid constraints that can be bypassed with brute force, hardcoded tables, or I/O tricks.
4. **Promote Alternative Paradigms**: Encourage different algorithmic patterns (e.g., two-pointers, heaps, prefix sums, monotonic stacks, union-find, DP, bit tricks) or different programming styles (iterative vs recursive, in-place vs extra space, streaming vs materialization).
5. **Verifiable**: Provide clear, automatable verification hints (keywords/regex/complexity target) rather than vague style suggestions.

## Recommended Constraint Archetypes (pick appropriately)
- Ban core library shortcuts (e.g., sorting APIs, Counter, regex, built-in min/max/sum/abs).
- Restrict access patterns (no direct indexing/slicing/comprehensions; streaming/iterator-only).
- Complexity goals (time O(n) instead of O(n log n); memory O(1) extra).
- Enumeration restrictions (no nested pair enumeration/Cartesian products; no recursion or no loops — choose one, not both).
- Data-structure shifts (avoid arrays-only; require or forbid stacks/heaps/queues/sets/maps as fits the problem).
- Paradigm shifts (no recursion; must use iterative with explicit stack; or the converse, depending on baseline).
- Side-effect/I/O restrictions (pure function; no globals; no printing; deterministic order without sorting).
